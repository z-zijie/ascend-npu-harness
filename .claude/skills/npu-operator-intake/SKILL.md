# npu-operator-intake

## Skill Name

`npu-operator-intake` -- Operator Requirement Intake

## Trigger Description

The user describes a new NPU operator in natural language. Examples:

- "Implement fused SwiGLU + quantization for shape [B,S,H], dtype fp16 input, int8 output, per-channel scales."
- "I need a GELU activation kernel."
- "Add a layer norm operator with eps=1e-5."

This skill is the first gate in the operator development pipeline. It extracts and normalizes the user's intent into a structured, machine-readable intake YAML.

## When to Use

- The user submits a new operator requirement (verbally or via `/npu-new-op`).
- The user mentions an operator name, math formula, or input/output shapes and it is not yet in `OPERATOR_REGISTRY.yaml`.
- An existing operator's requirements have changed and the intake YAML needs to be regenerated from updated natural-language input.

## When Not to Use

- The operator already has a valid intake YAML and a SPEC.yaml in the registry; use `npu-spec-authoring` instead.
- The user is asking about existing operators, debugging failures, or running tests -- the intake gate is only for new or substantially revised operators.
- The user is brainstorming and has not committed to creating a new operator -- ask the user to confirm intent first.

## Required Inputs

From the user's description, the following must be captured or explicitly recorded as "NOT SPECIFIED" with an assumption:

1. **Operator name** -- unique, snake_case, descriptive.
2. **Category** -- one of: `elementwise`, `reduction`, `layout`, `matmul`, `normalization`, `attention`, `fusion`, `custom`.
3. **Math definition** -- formula, equation, or prose description precise enough to implement.
4. **Inputs** -- each must have: name, dtype (or list of supported dtypes), shape (symbolic or concrete), layout (contiguous, NCHW, NHWC, ND), role (activation, weight, bias, scale, mask, position, scalar).
5. **Outputs** -- each must have: name, dtype, shape (with derivation rule from inputs), layout.
6. **Parameters** -- scalar parameters (e.g., eps, alpha, axis, rotary_dim) with type, default, and constraints.
7. **Shape rules** -- constraints on shape symbols (e.g., `rotary_dim <= H`, `rotary_dim % 2 == 0`).
8. **Dtype rules** -- accumulation width, promotion rules, output casting behavior.
9. **Layout rules** -- required layouts, supported layout permutations.
10. **Broadcast rules** -- whether broadcasting is needed and what forms are supported.
11. **Mask rules** -- mask types supported (none, causal, additive), masking behavior.
12. **Accuracy contract** -- per-dtype tolerances (abs and rel) or a statement that defaults apply.
13. **Performance contract** -- required/optional, target metric, target value, comparison operator, tuning budget.
14. **Target arch** -- always `dav-2201` unless the user explicitly overrides.

## Required Outputs

A single YAML file written to `.npu-harness/operators/<op_name>/INTAKE.yaml` with this minimum structure:

```yaml
# INTAKE.yaml -- generated by npu-operator-intake
intake_version: 1
generated_at: "<ISO timestamp>"

name: "<op_name>"
category: "<category>"
target_arch: "dav-2201"

math_definition: |
  <formula or prose>

inputs:
  - name: "<name>"
    dtype: ["<dtype1>", "<dtype2>"]
    shape: ["<sym1>", "<sym2>"]
    layout: "<layout>"
    role: "<role>"

outputs:
  - name: "<name>"
    dtype: ["<dtype1>"]
    shape: "<derivation rule>"
    layout: "<layout>"

parameters:
  - name: "<param>"
    type: "<float|int|bool>"
    default: <value>
    constraints: "<description>"

shape_symbols:
  <sym>:
    allowed: [<values>]
    min: <int>
    max: <int>

shape_rules:
  - "<rule>"

dtype_rules:
  accumulation: "<float32|same_as_input>"
  output_cast: "<specified_by_output_dtype|same_as_input>"

layout_rules:
  contiguous_required: <true|false>
  supported_layouts: ["<layouts>"]

broadcast_rules:
  supported: <true|false>
  forms: ["<forms>"]

mask_rules:
  supported: <true|false>
  types: ["<types>"]

accuracy_contract:
  float32: { abs: 1.0e-5, rel: 1.0e-5 }
  float16: { abs: 1.0e-2, rel: 1.0e-2 }

performance_contract:
  required: <true|false>
  targets:
    - shape: "<concrete>"
      dtype: "<dtype>"
      metric: "<latency_us|bandwidth_gb_s|tflops>"
      target: <value>
      comparison: "<<=|>=|==>"
  tuning_budget:
    max_iterations: 5

assumptions:
  - "<assumption 1>"
  - "<assumption 2>"

open_questions: []
```

## Step-by-Step Procedure

### Step 1: Parse the User Description

Read the user's description carefully. Extract every claim about the operator: name, formula, inputs, outputs, shapes, dtypes, parameters, constraints. Do not invent details. Do not guess shapes from common knowledge alone -- if the user does not state a shape or dtype, mark it as missing.

### Step 2: Classify the Operator Category

Assign one of the eight canonical categories based on the dominant computation:

| Category | Dominant computation |
|---|---|
| `elementwise` | Pointwise math (add, mul, gelu, silu, swish). |
| `reduction` | Reduction over one or more axes (sum, mean, max). |
| `layout` | Data layout transformation (transpose, permute, reshape). |
| `matmul` | Matrix multiplication, possibly with bias. |
| `normalization` | Normalization over a dimension (layer_norm, rms_norm, batch_norm). |
| `attention` | Attention mechanism (flash attention, multi-head attention). |
| `fusion` | Two or more distinct operations fused into one kernel. |
| `custom` | Does not fit the categories above. Use sparingly. |

### Step 3: Identify Missing Information

Check the intake completeness against this checklist. For each missing item, decide:

- **Can I safely infer it from the operator category or common practice?** If yes, record the assumption in the `assumptions` field.
- **Is it ambiguous and high-impact (shapes, dtypes, output definition)?** If yes, ask the user. Do not proceed without answers.
- **Is it ambiguous but low-impact (exact tolerance, exact benchmark shape)?** If yes, apply a reasonable default and record the assumption.

The following are **blocking deficiencies** -- the skill MUST reject the intake and ask the user:

- No output definition (shape and dtype).
- No shape information for any input (symbolic or concrete).
- No dtype specification for inputs or outputs.
- Operator name conflicts with an existing entry in `OPERATOR_REGISTRY.yaml`.

The following are **non-blocking deficiencies** -- apply defaults and record assumptions:

- Accuracy tolerances not specified -> use harness defaults (fp32 abs=1e-5 rel=1e-5, fp16 abs=1e-2 rel=1e-2).
- Performance target not specified -> mark `performance_contract.required: false`.
- Layout not specified -> default to `contiguous`.
- Broadcast rules not specified -> mark `broadcast_rules.supported: false`.
- Mask rules not specified -> mark `mask_rules.supported: false`.
- Tuning budget not specified -> default to `max_iterations: 5`.

### Step 4: Normalize Shapes to Symbolic Form

Convert user-provided concrete shapes into symbolic form using standard LLM symbols:

- `B` = batch size
- `S` / `S_q` / `S_k` = sequence lengths
- `H` = hidden dimension (model dimension for attention)
- `D` / `D_v` = head dimension / value head dimension
- `C` = channels
- `M`, `N`, `K` = generic matrix dimensions
- `rotary_dim` = RoPE rotary dimension

Add each symbol to `shape_symbols` with `allowed` values or `min`/`max` ranges.

### Step 5: Validate Shape Derivation Rules

For each output, write a precise derivation rule. Examples:

- `"same as input x"` for elementwise.
- `"[B, S, H]" where B, S, H are from input x` for normalization.
- `"[M, N]" where M = dim(x, 0), N = dim(y, 1)` for matmul.
- `"last dimension reduced: [B, S]" where input is [B, S, H]` for reduce_sum_lastdim.

### Step 6: Assign Accuracy and Performance Contracts

Accuracy:
- For elementwise operators: default fp32 tolerances.
- For reductions: tighter fp32, looser fp16 due to accumulation error.
- For matmul/attention: looser fp16 tolerances.
- Always use the harness defaults unless the user specifies otherwise.

Performance:
- If the user gives a concrete performance target, copy it into `performance_contract.targets`.
- If the user says "fast" or "optimize" but gives no number, mark `performance_contract.required: false` and note the assumption.
- If the user says nothing about performance, mark `performance_contract.required: false`.

### Step 7: Write the INTAKE.yaml

Write the completed YAML to `.npu-harness/operators/<op_name>/INTAKE.yaml`. Create the directory if it does not exist.

The YAML must be valid, parseable, and follow the structure shown in Required Outputs.

### Step 8: Present to the User

Show the user a human-readable summary:

```
Operator: <name>
Category: <category>
Inputs:  <count> tensors <list names>
Outputs: <count> tensors <list names>
Math:    <one-line summary>
Assumptions: <count> recorded
Open questions: <count> remaining (or "None -- ready for spec authoring")
```

If there are open questions, present each one concisely and wait for answers. If there are none, tell the user the intake is ready and the next step is `/npu-discuss-op <name>` or `npu-spec-authoring`.

### Step 9: Update OPERATOR_REGISTRY.yaml

Add or update the operator entry with status `intake_complete`:

```yaml
operators:
  <op_name>:
    spec: "specs/<op_name>.yaml"
    intake: ".npu-harness/operators/<op_name>/INTAKE.yaml"
    category: "<category>"
    intake_complete: true
    generated: false
    built_host: false
    tested_host: false
```

Do not overwrite existing fields that were set by later pipeline stages.

## Verification Checklist

Before declaring the intake complete, verify each item:

- [ ] Operator name is unique in `OPERATOR_REGISTRY.yaml`.
- [ ] Category is one of the eight canonical categories.
- [ ] Math definition is written precisely enough to implement a CPU golden.
- [ ] Every input has a name, dtype list, shape list, layout, and role.
- [ ] Every output has a name, dtype list, shape derivation rule, and layout.
- [ ] All parameters have type, default, and constraints.
- [ ] All shape symbols appearing in shapes are defined in `shape_symbols`.
- [ ] Shape rules are explicit and do not contradict each other.
- [ ] Dtype rules specify accumulation width and output casting.
- [ ] Layout rules specify whether contiguous is required.
- [ ] Broadcast rules state supported/unsupported unambiguously.
- [ ] Mask rules state supported/unsupported unambiguously.
- [ ] Accuracy contract has abs and rel tolerances for every declared dtype.
- [ ] Performance contract is present (required true or false).
- [ ] Target arch is `dav-2201` (unless explicitly overridden by user).
- [ ] All assumptions are recorded explicitly in the `assumptions` list.
- [ ] Any open questions are recorded in the `open_questions` list.
- [ ] INTAKE.yaml file is valid YAML (parseable).
- [ ] OPERATOR_REGISTRY.yaml is updated with `intake_complete: true`.
- [ ] User has been presented with the summary and any open questions.
