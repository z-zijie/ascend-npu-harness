# /npu-plan-op

Create an implementation plan for the operator that can be executed in small, verifiable tasks.

## Usage

```
/npu-plan-op <op_name>
```

## Arguments

- `op_name` — The operator name as registered in `OPERATOR_REGISTRY.yaml`.

## Preconditions

- `.npu-harness/operators/<op_name>/SPEC.yaml` exists, complete, and validated (no TODO/TBD/empty critical fields)
- `.npu-harness/operators/<op_name>/DECISIONS.md` exists
- Implementation scaffold exists under `ops/<op_name>/` (generated by `/npu-new-op`)
- Test file exists at `tests/test_<op_name>.cpp`

If the spec is not complete, run `/npu-discuss-op <op_name>` first.

## What to Read Before Planning

1. `.npu-harness/operators/<op_name>/SPEC.yaml` — Complete operator contract
2. `.npu-harness/operators/<op_name>/DECISIONS.md` — All decisions and assumptions
3. `ops/<op_name>/` — Current scaffold state
4. `tests/test_<op_name>.cpp` — Current test file state
5. `.npu-harness/config.yaml` — Harness config for defaults
6. Existing similar operators under `ops/` — Patterns and conventions to follow

## Plan Structure

The plan must be written to `.npu-harness/operators/<op_name>/PLAN.md` and include ALL of the following sections.

### 1. Operator Summary

```markdown
## Summary

- **Name:** <op_name>
- **Category:** <category>
- **Math:** <one-line formula>
- **Inputs:** <count> tensors + <count> scalars
- **Outputs:** <count> tensors
- **Dtypes:** <list>
- **Performance required:** <yes|no>
- **Target arch:** dav-2201
```

### 2. File Manifest

Every file that will be created or modified:

```markdown
## File Manifest

### New files
| File | Purpose |
|------|---------|
| ops/<op_name>/kernel/<op_name>_kernel.cpp | Ascend C kernel implementation |
| ops/<op_name>/host/<op_name>_tiling.cpp | Shape validation and tiling logic |
| ops/<op_name>/golden/<op_name>_golden.cpp | CPU reference implementation |
| ops/<op_name>/runner/<op_name>_runner.cpp | Host-side <<<>>> launch runner |
| ... | |

### Modified files
| File | Change |
|------|--------|
| ops/CMakeLists.txt | Add <op_name> subdirectory |
| tests/CMakeLists.txt | Add test_<op_name> target |
```

### 3. Implementation Steps

Numbered, ordered steps. Each step must be small enough to complete in one focused session. Each step must define its verification command.

```markdown
## Implementation Steps

### Step 1: Write failing tests
- **File:** tests/test_<op_name>.cpp
- **Action:** Implement GTest cases covering shapes, dtypes, edge cases
- **Expected:** Tests compile but fail (RED) — golden not yet implemented
- **Verify:** `cmake --build build/host && ctest --test-dir build/host -R <op_name> --output-on-failure`
- **Blocks on:** nothing

### Step 2: Implement CPU golden
- **File:** ops/<op_name>/golden/<op_name>_golden.cpp
- **Action:** Implement the reference C++ implementation
- **Expected:** Host tests pass (GREEN) — golden matches reference values
- **Verify:** `ctest --test-dir build/host -R <op_name> --output-on-failure`
- **Blocks on:** Step 1

### Step 3: Implement shape validation
- **File:** ops/<op_name>/host/<op_name>_tiling.cpp
- **Action:** Validate input shapes against spec rules; compute output shapes
- **Expected:** Invalid shapes rejected; valid shapes accepted
- **Verify:** `ctest --test-dir build/host -R <op_name> --output-on-failure`
- **Blocks on:** Step 2

### Step 4: Implement tiling
- **File:** ops/<op_name>/host/<op_name>_tiling.cpp
- **Action:** Compute tile sizes, block dimensions, alignment, tail policy
- **Expected:** Tiling validates for all test shapes
- **Verify:** `ctest --test-dir build/host -R <op_name> --output-on-failure`
- **Blocks on:** Step 3

### Step 5: Implement NPU kernel
- **File:** ops/<op_name>/kernel/<op_name>_kernel.cpp
- **Action:** Write the Ascend C kernel with proper block/tile processing
- **Expected:** Kernel compiles with ccec
- **Verify:** If CANN available: `cmake --build build/npu --target <op_name>_kernel`
- **Blocks on:** Step 4

### Step 6: Implement runner
- **File:** ops/<op_name>/runner/<op_name>_runner.cpp
- **Action:** Write the <<<>>> launch path (device alloc, copy, launch, sync, copy back)
- **Expected:** NPU tests pass
- **Verify:** If NPU available: `ctest --test-dir build/npu -R <op_name>_npu --output-on-failure`
- **Blocks on:** Step 5

### Step 7: Run full test suite
- **Action:** Run all host and NPU tests
- **Verify:** `ctest --test-dir build/host -R <op_name> --output-on-failure`
- **Verify (if NPU):** `ctest --test-dir build/npu -R <op_name> --output-on-failure`
- **Blocks on:** Steps 1-6

### Step 8: Accuracy report
- **Action:** Compare NPU output to CPU golden; produce accuracy.json
- **Verify:** `python3 scripts/harness.py verify <op_name>`
- **Blocks on:** Step 7
```

### 5. Expected Failures Before Implementation

List every test that should fail before implementation completes. This documents the RED phase.

```markdown
## Expected Failures (RED Phase)

| Test | Expected Behavior | Fixed By Step |
|------|-------------------|---------------|
| All host tests | Compile error / link error | Step 1 (scaffold) |
| Golden comparison | No golden to compare | Step 2 |
| Shape validation | Invalid shapes accepted | Step 3 |
| NPU kernel | No kernel compiled | Step 5 |
```

### 6. Verification Commands

```markdown
## Verification Commands

### Host-only
```bash
cmake -S . -B build/host -G Ninja -DHARNESS_ENABLE_ASCEND=OFF -DCMAKE_BUILD_TYPE=RelWithDebInfo
cmake --build build/host
ctest --test-dir build/host -R <op_name> --output-on-failure
```

### NPU (if available)
```bash
cmake -S . -B build/npu -G Ninja -DHARNESS_ENABLE_ASCEND=ON -DHARNESS_NPU_ARCH=dav-2201 -DCMAKE_BUILD_TYPE=RelWithDebInfo
cmake --build build/npu
ctest --test-dir build/npu -R <op_name> --output-on-failure
```

### Accuracy
```bash
python3 scripts/harness.py verify <op_name>
```
```

### 7. Performance Plan (if performance.required: true)

```markdown
## Performance Plan

### Targets
| Shape | Dtype | Metric | Target | Comparison |
|-------|-------|--------|--------|------------|
| <shape> | <dtype> | latency_us | <value> | <= |

### Benchmark Setup
- Warmup iterations: 10
- Measurement iterations: 100
- Device: dav-2201

### Tuning Strategy
1. Baseline measurement with auto tiling
2. If gap exists, hypothesize bottleneck (memory bandwidth, compute, tiling overhead)
3. Adjust tiling parameters
4. Re-measure
5. Compare before/after
6. Record in PERF.md

### Stop Conditions
- Target passes: SUCCESS
- Tuning budget exhausted (max_iterations reached): write GAP REPORT
- Concrete blocker (missing API, hardware limitation): document in BLOCKERS.md

### Tuning Budget
- **Max iterations:** <from spec or default 5>
- **Current iteration:** 0
```

### 8. Blocker Detection

```markdown
## Blocker Detection

Before and during implementation, check for these blockers:

### Environment Blockers
- [ ] CANN toolkit not installed — host-only mode works, NPU skipped
- [ ] ccec compiler not found — cannot build NPU kernels
- [ ] No visible NPU device — NPU tests will skip at runtime
- [ ] GTest not found — cannot build tests
- [ ] Ninja not found — cannot build

### Dependency Blockers
- [ ] <header> not available in CANN — kernel implementation blocked
- [ ] <API> not supported in this CANN version — use fallback or stub
- [ ] <library> not found — link step will fail

### Implementation Blockers
- [ ] <blocker description> — <specific missing piece>

If any blocker is found, document it in .npu-harness/operators/<op_name>/BLOCKERS.md
```

### 9. CMake Integration

```markdown
## CMake Integration

### ops/<op_name>/CMakeLists.txt
- `harness_add_operator(<op_name>)` or equivalent
- Separate targets for: golden, tiling, kernel, runner, bench
- Conditional kernel compilation on `HARNESS_ENABLE_ASCEND`

### ops/CMakeLists.txt
- `add_subdirectory(<op_name>)`

### tests/CMakeLists.txt
- `harness_add_test(test_<op_name>)` with CTest labels:
  - `<op_name>`
  - `<category>` (elementwise, reduction, matmul, etc.)
  - `host`
  - `golden`
```

## Rules

1. Every step must have a verification command. No step is complete without evidence.
2. Steps must be ordered by dependency. No circular dependencies.
3. The CPU golden always comes before the NPU kernel.
4. Shape validation comes before tiling.
5. Tiling comes before the NPU kernel.
6. Tests are written to fail first (RED), then implementation makes them pass (GREEN).
7. Performance plans are required only if `performance.required: true` in the spec.
8. Every blocker must be specific. "Too complex" is never a valid blocker.

## Output

Write the complete plan to `.npu-harness/operators/<op_name>/PLAN.md`.

After writing, update `.npu-harness/operators/<op_name>/STATUS.md`:

```markdown
| plan | done | |
```

Print a summary:

```
Plan complete for <op_name>.

Implementation steps: <N>
Test shapes: <M>
Performance targets: <K>
Potential blockers: <list any found>

Plan written to: .npu-harness/operators/<op_name>/PLAN.md

Next command: /npu-execute-op <op_name>
```
